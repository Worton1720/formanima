## 1. Цель и назначение системы

### 1.1 Назначение

FORMANIMA — веб-приложение для трекинга личных целей и привычек, учёта питания (калорий и макронутриентов) и управления личными финансами с интегрированной системой геймификации. Трекинг целей, калорий и финансов являются равноценными первоклассными функциями системы. Приложение позволяет пользователям формировать устойчивые поведенческие паттерны посредством визуализации прогресса, системы очков опыта (XP), уровней, рангов и достижений.

### 1.2 Цель разработки

Целью разработки является создание единой платформы, объединяющей инструменты управления целями и привычками, учёта финансовых транзакций и бюджетов, мониторинга питания (КБЖУ) и статистического анализа прогресса. Геймификационная составляющая призвана повысить мотивацию пользователей к регулярному выполнению запланированных действий за счёт немедленного положительного подкрепления и долгосрочных наград. Приложение реализовано по технологии SPA (Single-Page Application) с поддержкой PWA (Progressive Web App), что обеспечивает возможность установки на устройство и работу в режиме офлайн.

### 1.3 Область применения

Система предназначена для использования в качестве персонального инструмента продуктивности в среде настольного или мобильного браузера. Доступ к системе осуществляется через веб-интерфейс. Благодаря поддержке PWA приложение может быть установлено на домашний экран мобильного устройства или на рабочий стол ПК и запускаться как нативное приложение без открытия браузера. Статические ресурсы кэшируются при первом визите, что обеспечивает быстрый запуск и частичную работу в условиях нестабильного соединения.

## 2. Целевая аудитория

### 2.1 Основные группы пользователей

**Конечные пользователи (роль `user`)** — физические лица, стремящиеся к формированию и поддержанию полезных привычек, достижению личных целей, контролю бюджета и рациона питания. Предполагается базовая цифровая грамотность; специальных технических знаний не требуется.

**Администраторы системы (роль `admin`)** — лица, ответственные за техническую эксплуатацию платформы: управление учётными записями пользователей, просмотр системной статистики, блокировка недобросовестных пользователей.

## 5. Ограничения и допущения

### 5.1 Технические ограничения

1. **Платформа развёртывания.** Для локального запуска и разработки используется Docker Compose (версия 3.9). Производственное развёртывание предполагает наличие среды, совместимой с Docker. Запуск без Docker возможен при ручной настройке переменных окружения и подключения к PostgreSQL.

2. **СУБД.** Единственной поддерживаемой системой управления базами данных является PostgreSQL (версия 16). Использование иных СУБД (MySQL, SQLite, MongoDB и т. д.) не предусмотрено без рефакторинга ORM-схемы. Ряд особенностей реализации (в частности, поведение `NULL != NULL` в уникальных индексах при поиске по `GoalProgress.milestoneId`) привязан к семантике PostgreSQL.

3. **Мобильный доступ через PWA.** Нативные приложения для iOS и Android в рамках данной версии не разрабатываются. Вместо этого приложение поддерживает технологию PWA: пользователи могут установить его на домашний экран смартфона и получить app-like опыт (полноэкранный режим, кэширование, быстрый запуск) без публикации в App Store или Google Play.

4. **Однопользовательская аренда (single-tenant).** Архитектура системы рассчитана на одну организацию (или частного пользователя). Мультитенантность, предполагающая жёсткое разделение данных между организациями на уровне схемы или базы данных, не реализована.

5. **Отсутствие интеграции со сторонними сервисами.** В текущей версии не предусмотрены интеграции с внешними API: банковскими системами, фитнес-трекерами, системами единого входа (SSO/OAuth).

6. **Ограничения хранилища файлов.** Загрузка пользовательских файлов (аватары, документы) не поддерживается. Все данные хранятся в реляционной базе данных.

### 5.2 Допущения

1. Пользователи имеют доступ к браузеру с поддержкой ES2020 и включённым JavaScript.
2. Сетевое соединение между браузером клиента и сервером стабильно; обработка длительных перебоев в работе сети выходит за рамки данной версии.
3. Серверные переменные окружения (`JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, `DATABASE_URL` и др.) настраиваются при развёртывании; значения по умолчанию предназначены исключительно для среды разработки.
4. Часовой пояс для вычисления серий (streak) и «идеальных дней» геймификации совпадает с серверным UTC; локализация по часовым поясам пользователей не реализована.
5. Логика достижений является статической и определяется в конфигурационном файле `achievements.config.ts` на стороне сервера; добавление новых достижений требует изменения кода и пересборки сервера, но не требует миграции базы данных.
