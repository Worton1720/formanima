## 1. Общая архитектура системы

### 1.1. Описание архитектурного подхода

Система FORMANIMA реализована по классической двухзвенной архитектуре «клиент — сервер» с явным разделением ответственности между уровнями приложения. Клиентская часть представляет собой одностраничное приложение (SPA — Single-Page Application), реализованное как прогрессивное веб-приложение (PWA — Progressive Web App), которое взаимодействует с серверной частью исключительно через RESTful HTTP API. PWA-слой обеспечивает возможность установки приложения на устройство, кэширование статических ресурсов и частичную работу при нестабильном соединении. Серверная часть обеспечивает бизнес-логику, аутентификацию, работу с базой данных и вычисление показателей геймификации.

Подобное разделение обеспечивает следующие архитектурные преимущества:

- **Независимость развёртывания** — клиент и сервер могут масштабироваться и обновляться независимо друг от друга;
- **Чёткость контракта** — все взаимодействия определяются через HTTP-интерфейс с задокументированными схемами данных (OpenAPI/Swagger);
- **Тестируемость** — каждый уровень тестируется изолированно без зависимости от соседних слоёв;
- **Технологическая гибкость** — при необходимости клиентская часть может быть заменена на мобильное приложение без изменения сервера.

### 1.2. Уровни системы

Система состоит из четырёх логических уровней:

| Уровень | Технология | Назначение |
|---|---|---|
| Клиентский интерфейс | Vue 3 + Tailwind CSS v4 + PWA (vite-plugin-pwa) | Отображение данных, взаимодействие с пользователем, установка как приложения |
| Управление состоянием | Pinia | Хранение и синхронизация данных на клиенте |
| Сервер приложений | NestJS (Node.js) | Бизнес-логика, REST API, аутентификация |
| Хранилище данных | PostgreSQL + Prisma ORM | Персистентное хранение всех данных системы |

### 1.4. Инфраструктура и развёртывание

Вся инфраструктура описана в файле `docker-compose.yml`, что обеспечивает воспроизводимость среды и упрощает развёртывание на любой платформе. Сервисы разворачиваются в изолированных Docker-контейнерах и взаимодействуют через внутреннюю сеть Docker.

```
docker-compose.yml
├── service: postgres   — база данных PostgreSQL
├── service: server     — NestJS сервер приложений (порт 3000)
└── service: client     — Vue 3 SPA dev-сервер (порт 5173)
```

## 2. Серверная часть

### 2.1. Начальная конфигурация сервера

При запуске серверного приложения (`bootstrap`) выполняется глобальная конфигурация:

- устанавливается глобальный префикс маршрутов `/api`;
- подключается `ValidationPipe` с параметрами `whitelist: true` и `transform: true` — все входящие данные автоматически валидируются и приводятся к типам DTO;
- инициализируется Swagger UI по адресу `/api/docs` с описанием всех эндпоинтов, схем и требований к авторизации;
- настраивается CORS с разрешённым источником `http://localhost:5173`;
- приложение запускается на порту `3000`.

### 2.2. Модули сервера

Сервер организован по принципу модульности NestJS. Каждый модуль инкапсулирует контроллер, сервис и DTO-объекты конкретной предметной области.

| Модуль | Ответственность | Основные эндпоинты |
|---|---|---|
| **PrismaModule** | Глобальный провайдер подключения к БД через Prisma ORM | — (глобальный сервис) |
| **AuthModule** | Регистрация, вход, выход, обновление токенов, профиль пользователя | `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/logout`, `GET /auth/me`, `PATCH /auth/profile` |
| **GoalsModule** | CRUD целей и привычек, управление прогрессом и вехами | `GET/POST /goals`, `GET/PATCH/DELETE /goals/:id`, `POST /goals/:id/progress`, `GET/POST/PATCH /goals/:id/milestones` |
| **GamificationModule** | Расчёт XP, уровней, рангов, разблокировка достижений | `GET /gamification/rank`, `GET /gamification/achievements`, `POST /gamification/recalculate` |
| **FinanceModule** | Управление транзакциями, бюджетами, целями накоплений | `GET/POST /finance/transactions`, `GET/POST/PATCH /finance/budgets`, `GET/POST /finance/savings-goals` |
| **CaloriesModule** | Учёт питания, управление профилем калорийности | `GET/POST /calories/entries`, `GET/PATCH /calories/profile` |
| **StatsModule** | Агрегированная статистика по всем модулям | `GET /stats/summary`, `GET /stats/habits`, `GET /stats/finance`, `GET /stats/calories` |
| **AdminModule** | Административные операции, управление пользователями | `GET /admin/users`, `PATCH /admin/users/:id`, `DELETE /admin/users/:id` |

### 2.3. Структура каждого модуля

Каждый функциональный модуль организован по единому шаблону:

```
src/<module>/
├── <module>.module.ts       — объявление модуля NestJS
├── <module>.controller.ts   — HTTP-обработчики, маршруты
├── <module>.service.ts      — бизнес-логика
├── <module>.service.spec.ts — юнит-тесты сервиса
└── dto/
    ├── create-<entity>.dto.ts
    └── update-<entity>.dto.ts
```

Данная структура обеспечивает единообразие кодовой базы и облегчает ориентацию в проекте при его дальнейшем сопровождении.

### 2.4. Схема аутентификации (JWT-flow)

Аутентификация построена на двухтокенной схеме с использованием JWT (JSON Web Tokens).

**Параметры токенов:**

| Токен | Время жизни | Хранение | Назначение |
|---|---|---|---|
| Access token | 15 минут | Память клиента (Pinia store) | Авторизация запросов к API |
| Refresh token | 7 суток | БД (bcrypt-хэш), httpOnly cookie | Получение новой пары токенов |

**Схема взаимодействия:**

```
Клиент                          Сервер                        БД
  │                               │                            │
  │─── POST /auth/register ───────►│                            │
  │    { email, password, name }   │── bcrypt(password) ───────►│
  │                                │   INSERT User              │
  │◄── { accessToken, user } ─────│◄──────────────────────────-│
  │                               │                            │
  │─── POST /auth/login ──────────►│                            │
  │    { email, password }         │── SELECT User ────────────►│
  │                                │◄── user row ───────────────│
  │                                │── bcrypt.compare()         │
  │                                │── sign(accessToken 15m)    │
  │                                │── sign(refreshToken 7d)    │
  │                                │── bcrypt(refreshToken)     │
  │                                │── UPDATE User.refreshToken►│
  │◄── { accessToken, user } ─────│                            │
  │    Set-Cookie: refreshToken    │                            │
  │                               │                            │
  │─── GET /goals (+ Bearer) ─────►│                            │
  │                                │── JwtAuthGuard.verify()    │
  │◄── [goals array] ─────────────│                            │
  │                               │                            │
  │   [access token истёк]        │                            │
  │                               │                            │
  │─── POST /auth/refresh ────────►│                            │
  │    Cookie: refreshToken        │── SELECT User ────────────►│
  │                                │◄── user.refreshToken ──────│
  │                                │── bcrypt.compare()         │
  │                                │── sign(новый accessToken)  │
  │◄── { accessToken } ───────────│                            │
  │                               │                            │
  │─── POST /auth/logout ─────────►│                            │
  │                                │── UPDATE User:             │
  │                                │   refreshToken = null ─────►│
  │◄── 200 OK ─────────────────── │                            │
```

**Глобальная защита маршрутов:**

Декоратор `JwtAuthGuard` применяется глобально ко всем маршрутам. Для публичных эндпоинтов (регистрация, вход, обновление токена) используется декоратор `@Public()`, который исключает их из-под действия глобального guard'а. Все остальные маршруты требуют действительного access-токена в заголовке `Authorization: Bearer <token>`.

Административные маршруты дополнительно защищены проверкой поля `role === 'ADMIN'` в токене.

## 3. Схема базы данных

### 3.1. Описание моделей данных

База данных включает 11 основных моделей, организованных вокруг центральной сущности `User`.

#### 3.1.1. Модель User

Центральная сущность системы. Хранит учётные данные и связана со всеми остальными моделями.

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|
| id | String | PK, cuid() | Уникальный идентификатор |
| email | String | UNIQUE, NOT NULL | Электронная почта (логин) |
| passwordHash | String | NOT NULL | Хэш пароля (bcrypt) |
| name | String | NOT NULL | Отображаемое имя |
| refreshToken | String? | NULL | Хэш refresh-токена |
| role | Role | DEFAULT USER | Роль: USER или ADMIN |
| createdAt | DateTime | DEFAULT now() | Дата регистрации |
| updatedAt | DateTime | auto-update | Дата последнего обновления |

#### 3.1.2. Модель UserRank

Хранит игровые показатели пользователя. Связана с User отношением один-к-одному.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id (UNIQUE) |
| xp | Int | Текущее количество очков опыта |
| level | Int | Текущий уровень пользователя |
| totalStrikes | Int | Общее количество выполненных целей |
| rank | RankEnum | Ранг: apprentice / journeyman / master / grandmaster |

#### 3.1.3. Модель UserAchievement

Таблица разблокированных достижений пользователя (связующая таблица).

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id |
| achievementId | String | Идентификатор достижения (ACH-001..ACH-012) |
| unlockedAt | DateTime | Дата и время разблокировки |

#### 3.1.4. Модель Goal

Основная сущность трекера привычек и целей.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id |
| title | String | Название цели |
| description | String? | Описание |
| type | GoalType | Тип: числовой / булевый / накопительный и др. |
| status | GoalStatus | Статус: active / completed / archived / paused |
| category | String? | Категория цели |
| color | String? | Цвет для отображения |
| icon | String? | Иконка для отображения |
| targetValue | Float? | Целевое значение |
| currentValue | Float | Текущее значение (DEFAULT 0) |
| frequency | String? | Частота выполнения |
| deadline | DateTime? | Срок достижения цели |

#### 3.1.5. Модель GoalProgress

Журнал записей прогресса по цели.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| goalId | String | FK → Goal.id |
| userId | String | FK → User.id |
| milestoneId | String? | FK → Milestone.id (опционально) |
| date | DateTime | Дата записи |
| value | Float | Значение прогресса |
| note | String? | Заметка |
| completedAt | DateTime? | Время завершения |

#### 3.1.6. Модель Milestone

Промежуточные вехи на пути к достижению цели.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| goalId | String | FK → Goal.id |
| title | String | Название вехи |
| description | String? | Описание |
| order | Int | Порядковый номер |
| isDone | Boolean | Признак выполнения |
| doneAt | DateTime? | Дата выполнения |

#### 3.1.7. Модель Transaction

Финансовые транзакции пользователя.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id |
| type | TransactionType | income / expense |
| amount | Decimal | Сумма |
| category | String | Категория расхода/дохода |
| description | String? | Описание |
| date | DateTime | Дата транзакции |
| isRecurring | Boolean | Является ли повторяющейся |
| recurringInterval | String? | Интервал повторения |
| recurringParentId | String? | FK → Transaction.id (родительская транзакция) |

#### 3.1.8. Модель Budget

Бюджеты пользователя по категориям и месяцам.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id |
| category | String | Категория бюджета |
| month | String | Месяц в формате YYYY-MM |
| amount | Decimal | Плановая сумма бюджета |

**Уникальное ограничение:** `(userId, category, month)` — для каждой пары «пользователь + категория + месяц» допускается только одна запись бюджета.

#### 3.1.9. Модель SavingsGoal

Цели накоплений, опционально связанные с целью (Goal).

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id |
| title | String | Название цели накоплений |
| targetAmount | Decimal | Целевая сумма |
| currentAmount | Decimal | Текущая накопленная сумма |
| deadline | DateTime? | Срок достижения |
| goalId | String? | FK → Goal.id (связь с трекером целей) |

#### 3.1.10. Модель FoodEntry

Записи дневника питания.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id |
| name | String | Название блюда / продукта |
| mealType | MealType | breakfast / lunch / dinner / snack |
| calories | Float | Калорийность |
| protein | Float? | Белки (г) |
| fat | Float? | Жиры (г) |
| carbs | Float? | Углеводы (г) |
| weight | Float? | Вес порции (г) |
| date | DateTime | Дата приёма пищи |

#### 3.1.11. Модель CalorieProfile

Целевые показатели питания пользователя. Связана с User отношением один-к-одному.

| Поле | Тип | Описание |
|---|---|---|
| id | String | PK |
| userId | String | FK → User.id (UNIQUE) |
| targetCalories | Float | Целевая суточная калорийность |
| targetProtein | Float? | Целевое потребление белков (г) |
| targetFat | Float? | Целевое потребление жиров (г) |
| targetCarbs | Float? | Целевое потребление углеводов (г) |

## 4. Клиентская часть

### 4.1. Архитектура слоёв

Клиентское приложение организовано по принципу направленного потока данных с чётким разделением ответственности на четыре слоя.

```
┌─────────────────────────────────────────────────────────────────┐
│                     СЛОЙ ПРЕДСТАВЛЕНИЯ                          │
│                                                                 │
│   src/pages/*.vue          src/components/                      │
│   Страницы-маршруты        Переиспользуемые UI-компоненты       │
│                                                                 │
│   Отвечает за: рендеринг данных, обработку пользовательского    │
│   ввода, маршрутизацию, отображение состояний загрузки/ошибок   │
└──────────────────────────────┬──────────────────────────────────┘
                               │ читает состояние / вызывает actions
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                     СЛОЙ УПРАВЛЕНИЯ СОСТОЯНИЕМ                  │
│                                                                 │
│   src/stores/*.store.ts  (Pinia)                                │
```

### 4.2. Конфигурация axios

Axios-клиент настроен с помощью перехватчиков (interceptors):

- **Request interceptor** — автоматически добавляет заголовок `Authorization: Bearer <token>` к каждому исходящему запросу, считывая токен из `auth.store`;
- **Response interceptor** — при получении ответа `401 Unauthorized` автоматически вызывает `POST /auth/refresh`, получает новый access-токен, сохраняет его в store и повторяет исходный запрос с обновлённым токеном.

```
beforeEach(to, from):

    если to.meta.requiresAdmin и role !== ADMIN:

      redirect → /dashboard
```

**Service Worker** регистрируется автоматически при сборке и реализует две стратегии кэширования:

## 5. Система геймификации

### 5.1. Принципы системы

Система геймификации разработана для повышения мотивации пользователей к выполнению целей и привычек. Она основана на трёх взаимосвязанных механиках: накопление очков опыта (XP), повышение уровня и присвоение ранга.

### 5.2. Математические формулы

**Расчёт очков опыта (XP):**

```
XP = (totalStrikes × 10) + (perfectDayCount × 50) + Σ(achievementXP)
```

где:
- `totalStrikes` — суммарное количество выполненных целей за всё время;
- `perfectDayCount` — количество «идеальных дней» (все запланированные цели выполнены);
- `achievementXP` — бонусный XP от каждого разблокированного достижения.

**Расчёт уровня:**

```
Level = floor(√(XP / 100)) + 1
```

Данная формула обеспечивает нелинейный рост: начальные уровни достигаются быстро, а высокие уровни требуют значительно большего вклада, что поддерживает долгосрочный интерес пользователя.

**Примеры значений:**

| XP | Уровень |
|---|---|
| 0 | 1 |
| 100 | 2 |
| 400 | 3 |
| 900 | 4 |
| 2500 | 6 |
| 10000 | 11 |

### 5.3. Система рангов

Ранг определяется на основе суммарного количества ударов (totalStrikes):

| Ранг | Идентификатор | Порог (totalStrikes) |
|---|---|---|
| Ученик | `apprentice` | 0 — 99 |
| Подмастерье | `journeyman` | 100 — 499 |
| Мастер | `master` | 500 — 1499 |
| Грандмастер | `grandmaster` | ≥ 1500 |

Разделение прогресса на уровни (XP-based) и ранги (strikes-based) создаёт две независимые, но взаимодополняющие оси развития пользователя.

### 5.4. Таблица достижений

Система включает 12 достижений, охватывающих различные аспекты использования приложения:

| Код | Название | Условие разблокировки | Тип |
|---|---|---|---|
| ACH-001 | Первый шаг | Первое выполнение любой цели | Трекер целей |
| ACH-002 | Неделя без остановок | Серия 7 дней подряд | Трекер целей |
| ACH-003 | Месяц привычки | Серия 30 дней подряд | Трекер целей |
| ACH-004 | Марафонец | Серия 100 дней подряд | Трекер целей |
| ACH-005 | 500 шагов | Достижение 500 суммарных ударов | Трекер целей |
| ACH-006 | Тысячник | Достижение 1000 суммарных ударов | Трекер целей |
| ACH-007 | Легенда | Достижение 1500 суммарных ударов | Трекер целей |
| ACH-008 | Идеальная неделя | Все цели выполнены 7 дней подряд | Трекер целей |
| ACH-009 | Ранняя пташка | Выполнение цели до 8:00 | Особые условия |
| ACH-010 | Ночная сова | Выполнение цели после 23:00 | Особые условия |
| ACH-011 | Сытая серия | Серия записей питания 7 дней подряд | Калории |
| ACH-012 | Копилка | Достижение цели накоплений | Финансы |

### 5.5. Процедура пересчёта (recalculate)

При вызове `POST /gamification/recalculate` выполняется следующая последовательность шагов:

```
1. Подсчитать totalStrikes — сумму всех выполненных GoalProgress за всё время
2. Подсчитать perfectDayCount — дни, в которые все активные цели были выполнены
3. Получить список разблокированных достижений и сумму их XP
4. Вычислить XP по формуле
5. Вычислить Level по формуле
6. Определить Rank по порогам totalStrikes
7. Сохранить результат в UserRank (UPDATE или CREATE)
8. Проверить условия новых достижений и разблокировать при выполнении (INSERT UserAchievement)
9. Вернуть обновлённый объект UserRank
```

## 6. Схема взаимодействия компонентов

### 6.1. Сквозной поток данных

Ниже представлена полная схема прохождения типичного пользовательского запроса от интерфейса до базы данных и обратно.

```
┌──────────────────────────────────────────────────────────────────────┐
│                         ТИПИЧНЫЙ ЗАПРОС                              │
│                  (пример: добавление прогресса цели)                 │
└──────────────────────────────────────────────────────────────────────┘

[1] Пользователь нажимает кнопку «Отметить выполнение» на GoalsPage.vue
         │
         ▼
[2] GoalsPage.vue вызывает action: goalsStore.addProgress(goalId, dto)
         │
         ▼
[3] goals.store.ts устанавливает loading = true
    вызывает: goalsApi.createProgress(goalId, dto)
         │
         ▼
[4] goals.api.ts выполняет:
    POST /api/goals/:id/progress
    Headers: Authorization: Bearer <accessToken>
    Body: { value, date, note }
         │
         ▼
[5] NestJS: Request Pipeline
    │── JwtAuthGuard.canActivate()  → верификация JWT
    │── ValidationPipe              → валидация DTO
    └── GoalsController.createProgress()
         │
         ▼
[6] GoalsService.createProgress(userId, goalId, dto)
    │── Prisma: INSERT GoalProgress
    │── Prisma: UPDATE Goal.currentValue
    │── вызов GamificationService.recalculate(userId)
         │
         ▼
[7] GamificationService.recalculate(userId)
    │── Prisma: COUNT GoalProgress (totalStrikes)
    │── Prisma: вычисление perfectDays
    │── пересчёт XP, Level, Rank
    │── Prisma: UPSERT UserRank
    └── Prisma: проверка и INSERT UserAchievement при необходимости
         │
         ▼
[8] Ответ возвращается через цепочку:
    GoalsService → GoalsController → HTTP Response (201 Created)
         │
         ▼
[9] goals.api.ts получает ответ, возвращает данные в store
         │
         ▼
[10] goals.store.ts:
     │── обновляет goals[] в реактивном состоянии
     │── устанавливает loading = false
     └── вызывает gamificationStore.fetchRank()
              │
              ▼
[11] gamification.store.ts обновляет rank/level/xp
              │
              ▼
[12] Vue реактивно обновляет DOM:
     │── GoalsPage показывает обновлённый прогресс
     └── компонент RankWidget отображает новый XP/уровень
```
