# Шпаргалка на руку — вопросы комиссии (ответ в одну фразу)

Полные ответы — в `defense-questions.md`. Здесь — суть для быстрого взгляда.

## A. Предметная область
1. **AS-IS = блокнот?** — Продукт для личного использования; «как есть» — это ручное ведение, система его автоматизирует (TO-BE).
2. **В чём проблема?** — Не «негде записать», а удержание: люди бросают за 2–3 недели, нет прогресса и мотивации.
3. **Откуда 66 дней?** — Лалли, UCL, 2010, рецензируемое исследование (96 чел., 12 недель); миф про 21 день — от хирурга Мольца.
4. **ЦА?** — Широкая: студенты, специалисты, все, кто следит за продуктивностью и здоровьем.
5. **Зачем финансы и питание?** — Ключевое отличие: четыре домена под одной геймификацией, «всё в одном».

## B. Геймификация
6. **Почему уровни труднее?** — Опыт растёт по квадрату: ⌊√(XP/100)⌋+1, замедляет прокачку ради длинной дистанции.
7. **Как считается XP?** — отметки×10 + идеальные_дни×50 + опыт за достижения.
8. **Ранги vs уровни?** — Уровень от опыта, ранг от числа отметок (подмастерье/ремесленник/мастер/гроссмейстер).
9. **Научная база?** — Петля привычки Дахигга, теория самодетерминации Деси–Райана, «Октализ» Чоу.
10. **Награды не убивают мотивацию?** — Да, есть эффект сверхоправдания; поэтому геймификация — поддержка на старте, цели выбирает сам пользователь.
11. **Достижения в БД?** — Нет, в конфиге; в базе только факт разблокировки. Новые — без миграции.
12. **Когда пересчёт?** — Единый `recalculate` на сервере после любой отметки; клиент очки не считает.

## C. Архитектура
13. **Архитектура?** — Трёхзвенка: Vue 3 SPA → REST/axios → NestJS → Prisma → PostgreSQL, всё в Docker.
14. **Почему этот стек?** — Открытый и бесплатный (импортозамещение), модульность NestJS, реляционные связи под PostgreSQL.
15. **Что такое Prisma?** — ORM: модели в одной схеме, типобезопасные запросы и миграции, без ручного SQL.
16. **Почему REST, не GraphQL?** — Для CRUD проще и предсказуемее, легко документируется в Swagger (`/api/docs`).
17. **Цифры проекта?** — 8 модулей, 13 моделей, 54 эндпоинта, ≈8 600 строк (сервер 3 073 / клиент 5 535).
18. **Структура модуля?** — Контроллер + сервис + модуль + DTO + тесты; одинаково во всех 8 модулях.
19. **Валидация?** — Глобальный ValidationPipe + class-validator на DTO: отсекает лишнее, приводит типы.

## D. База данных
20. **13 таблиц, связи?** — В центре пользователь, вокруг 1:1 и 1:N; при удалении — каскад.
21. **Денормализация currentValue?** — Сумма отметок хранится в цели и обновляется при записи ради быстрого чтения.
22. **findFirst, не findUnique?** — milestoneId может быть NULL, а в PostgreSQL NULL ≠ NULL в уникальном индексе.
23. **Месяц строкой «ГГГГ-ММ»?** — Бюджет привязан к месяцу целиком, удобнее для группировки и ключа.

## E. Безопасность / ФЗ-152
24. **Защита ПДн?** — Хеш паролей bcrypt, токены JWT с обновлением, разграничение ролей.
25. **JWT, зачем два токена?** — access короткий для запросов, refresh длинный (хранится хешем, отзываемый) для тихого обновления.
26. **Истёк access?** — Клиент на 401 сам обновляет токен и повторяет запросы; пользователь не замечает.
27. **Права?** — Глобальный JwtAuthGuard + RolesGuard на админке; роль `blocked` отклоняется на входе.
28. **Соль bcrypt?** — Случайная добавка, чтобы одинаковые пароли давали разные хеши; фактор стоимости 10.
29. **SQL-инъекции?** — Прямого SQL нет, Prisma параметризует запросы.

## F. Тестирование
30. **Почему 33%?** — Покрыта основная логика (60–82%), не покрыты вспомогательные части; 48 тестов в 6 наборах, каркас заложен.
31. **Что тестируете первым?** — Движок геймификации: серии, идеальные дни, достижения, формулы.
32. **Без реальной БД?** — Через моки Prisma; тест проверяет логику, не СУБД.
33. **Ручное тестирование?** — Да, основные сценарии со скриншотами; результаты в `TEST_FINDINGS.md`.

## G. Аналоги
34. **Отличие от Habitica/Forest/Loop?** — Единственное объединяющее привычки+цели+финансы+питание, на русском, с self-hosting.
35. **Зачем self-hosting?** — Контроль над данными + демонстрация полного владения стеком (одна команда Docker).
36. **В чём новизна?** — Не геймификация сама по себе, а интеграция четырёх доменов под одной обоснованной моделью.

## H. Каверзные
37. **Самое сложное?** — Сделать геймификацию единой точкой пересчёта, вызываемой из всех модулей без задвоения очков.
38. **Что улучшить?** — Тесты до контроллеров, push-напоминания, социальные функции (драйвер «связанность»).
39. **Прототип или продукт?** — Рабочее приложение, все модули функционируют; технически готово к развёртыванию.
40. **Ваш вклад?** — Модель данных, архитектура, вся логика и оригинальный движок геймификации; фреймворки — готовые.
41. **Масштабируется?** — Да, сервер stateless; пересчёт при росте вынес бы в фоновую очередь.
42. **Почему счёт на сервере?** — Целостность и защита от накрутки: клиент можно подменить.
